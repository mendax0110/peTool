Dear Im\+Gui is a {\bfseries{bloat-\/free graphical user interface library for C++}}. It outputs optimized vertex buffers that you can render anytime in your 3D-\/pipeline-\/enabled application. It is fast, portable, renderer agnostic, and self-\/contained (no external dependencies).

Dear Im\+Gui is designed to {\bfseries{enable fast iterations}} and to {\bfseries{empower programmers}} to create {\bfseries{content creation tools and visualization / debug tools}} (as opposed to UI for the average end-\/user). It favors simplicity and productivity toward this goal and lacks certain features commonly found in more high-\/level libraries.

Dear Im\+Gui is particularly suited to integration in game engines (for tooling), real-\/time 3D applications, fullscreen applications, embedded applications, or any applications on console platforms where operating system features are non-\/standard.


\begin{DoxyItemize}
\item Minimize state synchronization.
\item Minimize UI-\/related state storage on user side.
\item Minimize setup and maintenance.
\item Easy to use to create dynamic UI which are the reflection of a dynamic data set.
\item Easy to use to create code-\/driven and data-\/driven tools.
\item Easy to use to create ad hoc short-\/lived tools and long-\/lived, more elaborate tools.
\item Easy to hack and improve.
\item Portable, minimize dependencies, run on target (consoles, phones, etc.).
\item Efficient runtime and memory consumption.
\item Battle-\/tested, used by \href{https://github.com/ocornut/imgui/wiki/Software-using-dear-imgui}{\texttt{ many major actors in the game industry}}.
\end{DoxyItemize}\hypertarget{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md162}{}\doxysubsection{\texorpdfstring{Usage}{Usage}}\label{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md162}
{\bfseries{The core of Dear Im\+Gui is self-\/contained within a few platform-\/agnostic files}} which you can easily compile in your application/engine. They are all the files in the root folder of the repository (imgui\texorpdfstring{$\ast$}{*}.cpp, imgui\texorpdfstring{$\ast$}{*}.h). {\bfseries{No specific build process is required}}. You can add the .cpp files into your existing project.

{\bfseries{Backends for a variety of graphics API and rendering platforms}} are provided in the \href{https://github.com/ocornut/imgui/tree/master/backends}{\texttt{ backends/}} folder, along with example applications in the \href{https://github.com/ocornut/imgui/tree/master/examples}{\texttt{ examples/}} folder. You may also create your own backend. Anywhere where you can render textured triangles, you can render Dear Im\+Gui.

See the \href{https://github.com/ocornut/imgui/wiki/Getting-Started}{\texttt{ Getting Started}} guide and Integration section of this document for more details.

Dear Im\+Gui allows you to {\bfseries{create elaborate tools}} as well as very short-\/lived ones. On the extreme side of short-\/livedness\+: using the Edit\&Continue (hot code reload) feature of modern compilers you can add a few widgets to tweak variables while your application is running, and remove the code a minute later! Dear Im\+Gui is not just for tweaking values. You can use it to trace a running algorithm by just emitting text commands. You can use it along with your own reflection data to browse your dataset live. You can use it to expose the internals of a subsystem in your engine, to create a logger, an inspection tool, a profiler, a debugger, an entire game-\/making editor/framework, etc.\hypertarget{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md163}{}\doxysubsection{\texorpdfstring{How it works}{How it works}}\label{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md163}
The IMGUI paradigm through its API tries to minimize superfluous state duplication, state synchronization, and state retention from the user\textquotesingle{}s point of view. It is less error-\/prone (less code and fewer bugs) than traditional retained-\/mode interfaces, and lends itself to creating dynamic user interfaces. Check out the Wiki\textquotesingle{}s \href{https://github.com/ocornut/imgui/wiki\#about-the-imgui-paradigm}{\texttt{ About the IMGUI paradigm}} section for more details.

Dear Im\+Gui outputs vertex buffers and command lists that you can easily render in your application. The number of draw calls and state changes required to render them is fairly small. Because Dear Im\+Gui doesn\textquotesingle{}t know or touch graphics state directly, you can call its functions anywhere in your code (e.\+g. in the middle of a running algorithm, or in the middle of your own rendering process). Refer to the sample applications in the examples/ folder for instructions on how to integrate Dear Im\+Gui with your existing codebase.

{\itshape A common misunderstanding is to mistake immediate mode GUI for immediate mode rendering, which usually implies hammering your driver/\+GPU with a bunch of inefficient draw calls and state changes as the GUI functions are called. This is NOT what Dear Im\+Gui does. Dear Im\+Gui outputs vertex buffers and a small list of draw calls batches. It never touches your GPU directly. The draw call batches are decently optimal and you can render them later, in your app or even remotely.}\hypertarget{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md164}{}\doxysubsection{\texorpdfstring{Releases \& Changelogs}{Releases \& Changelogs}}\label{md_external_2imgui_2docs_2_r_e_a_d_m_e_autotoc_md164}

See the \href{https://github.com/ocornut/imgui/wiki/Getting-Started}{\texttt{ Getting Started}} guide for details.

On most platforms and when using C++, {\bfseries{you should be able to use a combination of the \href{https://github.com/ocornut/imgui/tree/master/backends}{\texttt{ imgui\+\_\+impl\+\_\+xxxx}} backends without modification}} (e.\+g. {\ttfamily imgui\+\_\+impl\+\_\+win32.\+cpp} + {\ttfamily imgui\+\_\+impl\+\_\+dx11.\+cpp}). If your engine supports multiple platforms, consider using more imgui\+\_\+impl\+\_\+xxxx files instead of rewriting them\+: this will be less work for you, and you can get Dear Im\+Gui running immediately. You can {\itshape later} decide to rewrite a custom backend using your custom engine functions if you wish so.

Integrating Dear Im\+Gui within your custom engine is a matter of 1) wiring mouse/keyboard/gamepad inputs 2) uploading a texture to your GPU/render engine 3) providing a render function that can bind textures and render textured triangles, which is essentially what Backends are doing. The \href{https://github.com/ocornut/imgui/tree/master/examples}{\texttt{ examples/}} folder is populated with applications doing just that\+: setting up a window and using backends. If you follow the \href{https://github.com/ocornut/imgui/wiki/Getting-Started}{\texttt{ Getting Started}} guide it should in theory takes you less than an hour to integrate Dear Im\+Gui. {\bfseries{Make sure to spend time reading the \href{https://www.dearimgui.com/faq}{\texttt{ FAQ}}, comments, and the examples applications!}}

{\bfseries{Which version should I get?}}

We occasionally tag \href{https://github.com/ocornut/imgui/releases}{\texttt{ Releases}} (with nice releases notes) but it is generally safe and recommended to sync to latest {\ttfamily master} or {\ttfamily docking} branch.
